# وثيقة نظام QIB Service Portal

**اسم النظام:** QIB Service Portal  
**الجهة:** بنك القطيبي الإسلامي  
**نوع النظام:** بوابة داخلية لإدارة الطلبات، الموافقات، المراسلات، الوثائق، التقارير، والإعدادات التشغيلية  
**آخر تحديث للوثيقة:** 2026-05-11

---

## 1. نظرة عامة

نظام **QIB Service Portal** هو منصة داخلية موجهة لبيئة مصرفية، تهدف إلى توحيد استقبال الطلبات الداخلية، تمريرها عبر مسارات موافقات مرنة، تنفيذها من الأقسام المختصة، وتوثيق كل العمليات من خلال السجلات، التقارير، والمراسلات الداخلية.

النظام بدأ كنظام لإدارة طلبات تقنية المعلومات، ثم توسع ليصبح مناسباً للاستخدام على مستوى البنك، بحيث يمكن تعريف أنواع طلبات ومسارات موافقة مختلفة لكل إدارة أو وحدة.

أهم ما يوفره النظام:

- إدارة الطلبات الداخلية.
- بناء أنواع طلبات ديناميكية.
- مسارات موافقات قابلة للتخصيص.
- شاشة موافقات وتنفيذ.
- مراسلات داخلية مرتبطة بالطلبات.
- مكتبة وثائق PDF.
- إدارة مستخدمين وصلاحيات.
- تقارير وتشغيل PDF/Excel.
- إعدادات الذكاء الاصطناعي للمراسلات.
- إدارة قاعدة البيانات والنسخ الاحتياطي.
- مراقبة صحة النظام.
- سجل تدقيق للعمليات الحساسة.

## 4. تسجيل الدخول والجلسات

يدعم النظام حالياً تسجيل الدخول باستخدام البريد الإلكتروني وكلمة المرور.

آلية العمل:

1. المستخدم يدخل بياناته في شاشة الدخول.
2. الخلفية تتحقق من البريد وكلمة المرور.
3. عند النجاح يتم إصدار JWT Token.
4. الواجهة تحفظ التوكن في التخزين المحلي.
5. كل طلب API يرسل التوكن في ترويسة `Authorization`.

أهم نقاط الأمان:

- تشفير كلمات المرور.
- دعم إجبار المستخدم على تغيير كلمة المرور.
- دعم قفل أو تعطيل المستخدم.
- دعم إنهاء الجلسات.
- مراقبة صلاحيات الشاشات بعد الدخول.

مسارات API ذات صلة:

```text
POST /api/v1/auth/login
GET  /api/v1/auth/me
POST /api/v1/auth/logout
```

## 5. المستخدمون والصلاحيات

### 5.1 المستخدمون

تدعم شاشة المستخدمين:

- إضافة مستخدم.
- تعديل بيانات مستخدم.
- تعطيل أو تفعيل الحساب.
- قفل أو فك قفل الحساب.
- إعادة تعيين كلمة المرور.
- إجبار المستخدم على تغيير كلمة المرور.
- ربط المستخدم بإدارة.
- تحديد المدير المباشر.
- تحديد الدور.
- تحديد القسم المختص للمستخدم إن كان مختص معالجة.

### 5.2 الأدوار

الأدوار الأساسية المقترحة للنظام بعد توسيعه للبنك:

- موظف
- مدير مباشر
- مدير إدارة
- مختص معالجة
- مدير النظام

أدوار قديمة أو خاصة يمكن الاحتفاظ بها فقط عند الحاجة، لكن الأفضل ألا تستخدم كتصميم أساسي لكل البنك.

### 5.3 صلاحيات الشاشات

صلاحيات الشاشات تحدد قدرة المستخدم على دخول شاشة معينة ومستوى التعامل معها:

- لا يوجد
- عرض
- إضافة
- تعديل
- حذف
- تصدير
- إدارة

مثال:

```text
المستخدم جمال جلال لديه صلاحية "عرض" على شاشة الوثائق
=> يستطيع فتح الشاشة وعرض الوثائق المسموحة فقط.
```

### 5.4 الصلاحيات الحساسة

بعض الإجراءات يجب أن تبقى منفصلة عن صلاحيات الشاشات، مثل:

- إعادة ضبط قاعدة البيانات.
- استعادة نسخة احتياطية.
- إدارة إعدادات الذكاء الاصطناعي.
- عرض الرسائل السرية.
- حذف نهائي.
- تشغيل تحديثات النظام.

القاعدة الموصى بها:

```text
صلاحية الشاشة تسمح بالدخول والعمل العادي.
الصلاحية الحساسة تتحكم بالإجراءات الخطرة.
```

## 6. الطلبات

وحدة الطلبات هي قلب النظام.

المستخدم يستطيع:

- إنشاء طلب جديد.
- اختيار نوع الطلب.
- تعبئة الحقول الديناميكية.
- إرفاق ملفات عند الحاجة.
- متابعة حالة الطلب.
- تعديل الطلب إذا تم إرجاعه للتعديل.
- إعادة إرسال الطلب.
- طباعة الطلب PDF.
- إرسال مراسلة مرتبطة بالطلب إن كانت مفعلة.

### 6.1 دورة حياة الطلب

الحالات الأساسية:

```text
مسودة
مقدم
بانتظار الموافقة
معاد للتعديل
معتمد
قيد التنفيذ
مكتمل
مغلق
مرفوض
ملغي
```

### 6.2 إنشاء طلب

عند إنشاء طلب:

1. يتم تحميل أنواع الطلبات النشطة.
2. يختار المستخدم نوع الطلب.
3. يتم تحميل حقول النموذج من إعدادات نوع الطلب.
4. يتم تطبيق قواعد المرفقات.
5. يتم إنشاء نسخة Snapshot من بيانات النموذج ومسار الموافقات.
6. يبدأ الطلب من أول خطوة موافقة.

### 6.3 الحقول الديناميكية

كل نوع طلب يمكن أن يحتوي على حقول مختلفة، مثل:

- نص قصير
- نص طويل
- رقم
- تاريخ
- قائمة اختيار
- اختيار متعدد
- بريد إلكتروني
- رقم هاتف
- مستخدم
- إدارة
- ملف

البيانات المعروضة في تفاصيل الطلب يجب أن تعتمد على Snapshot وقت إنشاء الطلب، حتى لا تتأثر الطلبات القديمة عند تعديل نوع الطلب.

## 7. إدارة الطلبات

شاشة إدارة الطلبات تتحكم في:

- أنواع الطلبات.
- الحقول.
- مسارات الموافقة.
- قواعد المرفقات.
- SLA.
- الأولوية الافتراضية.
- القسم المختص.
- إعدادات الطباعة PDF.
- ربط المراسلات بالطلبات.

### 7.1 أنواع الطلبات

كل نوع طلب يحتوي على:

- اسم عربي.
- اسم إنجليزي.
- كود.
- وصف.
- قسم مختص.
- أولوية افتراضية.
- حالة التفعيل.
- هل يتطلب مرفقاً.
- هل يسمح بعدة مرفقات.
- قواعد SLA.

### 7.2 مسار الموافقات

يمكن بناء مسار لكل نوع طلب.

أنواع الخطوات:

- المدير المباشر.
- مدير إدارة محددة.
- مدير الإدارة المختصة.
- مختص الإدارة المختصة.
- دور محدد.
- مستخدم محدد.
- مختص تنفيذ.
- إغلاق الطلب.

مثال مسار:

```text
الموظف يقدم الطلب
=> المدير المباشر
=> مدير تقنية المعلومات
=> مختص قسم السيرفرات
=> مكتمل
```

### 7.3 قاعدة مهمة

تعديل مسار نوع طلب يؤثر على الطلبات الجديدة فقط.

الطلبات القديمة يجب أن تبقى على المسار الذي تم إنشاؤها به.

## 8. الموافقات والتنفيذ

شاشة الموافقات تعرض الطلبات حسب دور المستخدم وصلاحياته:

- بانتظار موافقتي.
- بانتظار التنفيذ.
- المعادة للتعديل.
- المتأخرة.
- المكتملة.
- سجل الموافقات.

القرارات الممكنة:

- موافقة.
- رفض.
- إرجاع للتعديل.
- تنفيذ.

قواعد مهمة:

- لا يستطيع المستخدم الموافقة إلا إذا كانت الخطوة الحالية مسندة له.
- إذا كانت الخطوة لا تسمح بالرفض لا يظهر زر الرفض، والخلفية تمنع الرفض أيضاً.
- إذا كانت الخطوة لا تسمح بالإرجاع للتعديل لا يظهر الزر، والخلفية تمنعه أيضاً.
- كل إجراء يتم تسجيله في سجل التدقيق.

## 9. المراسلات الداخلية

وحدة المراسلات هي بريد داخلي مرتبط بالنظام.

تدعم:

- الوارد.
- الصادر.
- غير المقروءة.
- المؤرشفة.
- المراسلات المرتبطة بالطلبات.
- الرسائل الرسمية.
- طلبات الاستيضاح.
- المسودات عند تفعيلها.

### 9.1 إنشاء رسالة

حقول الرسالة:

- المستلمون.
- نوع الرسالة.
- الأولوية.
- تصنيف السرية.
- الطلب المرتبط.
- الموضوع.
- نص الرسالة.
- المرفقات.

### 9.2 ربط الرسائل بالطلبات

عند ربط رسالة بطلب:

- تظهر في تبويب المراسلات داخل تفاصيل الطلب.
- تخضع لصلاحيات الرسائل وصلاحيات الطلب معاً.
- يمكن تضمين الرسائل الرسمية في PDF عند تفعيل الإعداد.

### 9.3 إعدادات المراسلات

يمكن التحكم في:

- تفعيل المراسلات.
- السماح بالرد.
- السماح بالتحويل.
- السماح بالأرشفة.
- السماح بالتعاميم.
- تحديد من يستطيع إرسال التعاميم.
- السماح بالمرفقات.
- الامتدادات المسموحة.
- الحد الأقصى لحجم المرفق.
- تصنيفات السرية.
- قوالب الرسائل.
- إعدادات المساعد الذكي داخل المراسلات.

## 10. مكتبة الوثائق

مكتبة الوثائق مخصصة لملفات PDF فقط.

تستخدم لحفظ:

- القرارات.
- التعاميم.
- البلاغات.
- النماذج.
- خدمات البنك.
- الهيكل التنظيمي.
- الخطط الرقمية.
- الخطط التشغيلية.
- الوثائق الخارجية.
- السياسات.
- التعليمات.
- الإجراءات.
- فتاوى الهيئة الشرعية.
- الوصف الوظيفي.

### 10.1 خصائص الوثيقة

كل وثيقة تحتوي على:

- عنوان عربي.
- عنوان إنجليزي اختياري.
- تصنيف.
- رقم وثيقة.
- وصف.
- الإدارة المالكة.
- درجة سرية.
- تاريخ إصدار.
- تاريخ سريان.
- تاريخ مراجعة.
- كلمات مفتاحية.
- إصدار حالي.
- هل تتطلب إقرار اطلاع.

### 10.2 الإصدارات

لا يتم تعديل ملف PDF القديم.

عند الحاجة لتغيير المحتوى:

```text
يتم رفع إصدار جديد.
يبقى الإصدار القديم محفوظاً.
يتم تعيين الإصدار الجديد كإصدار حالي.
```

### 10.3 الإقرار بالاطلاع

إذا كانت الوثيقة تتطلب إقرار اطلاع:

- يظهر زر "أقر بالاطلاع".
- يتم تسجيل المستخدم والوقت والإصدار.
- يمكن معرفة من أقر ومن لم يقر.

## 11. التقارير

شاشة التقارير تعرض مؤشرات حسب صلاحية المستخدم.

أنواع التقارير:

- تقارير الطلبات.
- تقارير الموافقات.
- تقارير SLA.
- تقارير المستخدمين والصلاحيات.
- تقارير المراسلات.
- تقارير المرفقات.
- تقارير التدقيق.

التصدير:

- Excel.
- PDF.

مبدأ مهم:

```text
التقارير يجب أن تحترم صلاحيات المستخدم ونطاق بياناته.
```

مثال:

- الموظف يرى طلباته فقط.
- المدير يرى طلباته وطلبات فريقه.
- مختص المعالجة يرى طلبات قسمه المختص.
- مدير النظام يرى كل شيء.

## 12. الذكاء الاصطناعي

النظام يدعم مساعداً ذكياً داخل المراسلات.

الهدف:

- توليد مسودة رسالة.
- تحسين الصياغة.
- جعل النص رسمياً.
- اختصار النص.
- اقتراح رد.
- تلخيص المراسلات.
- فحص المعلومات الناقصة.

قواعد أساسية:

- المساعد لا يرسل الرسائل تلقائياً.
- المساعد لا يعتمد أو يرفض الطلبات.
- المساعد لا يغير حالة أي طلب.
- المستخدم يجب أن يراجع النص ويضغط "استخدام النص" يدوياً.

### 12.1 إعدادات المزود

يدعم النظام مزوداً محلياً مثل Ollama أو API خارجي حسب الإعدادات.

الإعدادات تشمل:

- تفعيل المساعد.
- المزود.
- رابط API.
- اسم النموذج.
- الحد الأقصى لطول النص.
- مهلة الاستجابة.
- إخفاء البيانات الحساسة.
- System Prompt لتحديد وظيفة المساعد.

### 12.2 الخصوصية

قبل إرسال النص للنموذج يمكن إخفاء:

- البريد الإلكتروني.
- رقم الجوال.
- الرقم الوظيفي.
- أسماء المستخدمين.
- أرقام الطلبات.
- الرموز والمفاتيح وكلمات المرور.

## 13. قاعدة البيانات والنسخ الاحتياطي

شاشة قاعدة البيانات توفر:

- عرض حالة قاعدة البيانات.
- إنشاء نسخة احتياطية.
- تحميل نسخة احتياطية.
- التحقق من النسخة.
- استعادة نسخة.
- معاينة إعادة الضبط.
- صيانة قاعدة البيانات.
- فحص الجداول.
- عرض سجل العمليات.
- إعدادات النسخ الاحتياطي.

قواعد أمان مهمة:

- لا تتم الاستعادة مباشرة بعد الرفع.
- يجب التحقق من النسخة أولاً.
- يجب إنشاء نسخة احتياطية قبل الاستعادة أو إعادة الضبط.
- يجب عدم حذف مجلدات Docker volumes.
- يجب عدم حذف المرفقات إلا إذا تم اختيار ذلك صراحة.
- لا تعرض كلمة مرور قاعدة البيانات أو الاتصال الكامل.

## 14. مراقبة صحة النظام

- وجود مهام فاشلة.

## 18. الملفات والمرفقات

المرفقات تستخدم في:

- الطلبات.
- المراسلات.
- الوثائق.

القواعد العامة:

```text

```

## 19. السيناريوهات التشغيلية

### 19.1 طلب إنشاء سيرفر

مثال مسار:

```text
موظف يقدم طلب إنشاء سيرفر
=> المدير المباشر يعتمد
=> مدير تقنية المعلومات يعتمد
=> مختص قسم السيرفرات ينفذ
=> الطلب مكتمل
```

### 19.2 طلب ترخيص جهاز جديد

مثال مسار:

```text
موظف في إدارة الفروع يقدم الطلب
=> المدير المباشر
=> مدير وحدة التطوير والبرامج
=> مختص وحدة التطوير
=> مكتمل
```

### 19.3 وثيقة تتطلب إقرار اطلاع

```text
مدير الوثائق يرفع سياسة جديدة
=> يحدد أنها تتطلب إقرار اطلاع
=> المستخدمون المصرح لهم يفتحون الوثيقة
=> يضغط المستخدم "أقر بالاطلاع"
=> يسجل النظام الإقرار والوقت والإصدار
```

### 19.4 مراسلة مرتبطة بطلب

```text
مستخدم يفتح تفاصيل الطلب
=> يفتح تبويب المراسلات
=> يرسل طلب استيضاح مرتبط بالطلب
=> المستلم يرى الرسالة في الوارد
=> تظهر الرسالة ضمن مراسلات الطلب حسب الصلاحيات
```

## 23. ربط OpenID Connect مستقبلاً

يمكن ربط النظام مع OpenID Connect لتوفير SSO.

التصميم المقترح:

```text
المستخدم يضغط تسجيل الدخول عبر OpenID
=> يتم تحويله إلى مزود الهوية
=> يعود إلى /auth/oidc/callback
=> الخلفية تتحقق من ID Token
=> النظام يصدر JWT داخلي
=> الصلاحيات تبقى من قاعدة بيانات النظام
```

الأفضل أن يستخدم OpenID للتحقق من الهوية فقط، بينما تبقى صلاحيات الشاشات والطلبات داخل النظام.

## 26. خلاصة

QIB Service Portal هو نظام داخلي متكامل لإدارة العمل المؤسسي داخل البنك. قوته الأساسية في أنه يسمح بتعريف طلبات ومسارات موافقات ديناميكية، ويربط الطلبات بالمراسلات والوثائق والتقارير والتدقيق.

للحفاظ على جودة النظام مع توسع استخدامه على مستوى البنك، يجب التركيز على:

- وضوح الصلاحيات.
- بساطة شاشات الاستخدام اليومية.
- حماية البيانات.
- حفظ السجلات.
- ضمان انعكاس الإعدادات على السلوك الفعلي.
- مراقبة صحة النظام والنسخ الاحتياطي.
